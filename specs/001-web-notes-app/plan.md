## Summary

Build a personal web-based notes application with Markdown editing, hashtag organization, pinned notes, file attachments, PWA support, and dark/light theme. The backend is a Go REST API (chi router, modernc.org/sqlite) with frontend assets embedded via `//go:embed`. The frontend uses vanilla JS + EasyMDE for editing. The entire system runs as a single Docker container (~20-25MB image) on Alpine 3.19, with SQLite database and uploaded files stored on external volumes. A GitHub Actions workflow publishes the image to `ghcr.io/edalcin/` on every push to `main`.

## Technical Context

**Language/Version**: Go 1.23 (backend), ES2022 Vanilla JS (frontend)
**Primary Dependencies**: `chi` (HTTP router), `modernc.org/sqlite` (pure-Go SQLite driver), EasyMDE v2 (Markdown editor), Marked.js (Markdown renderer)
**Storage**: SQLite (external volume via `DB_PATH`), filesystem for files (external volume via `FILES_PATH`)
**Testing**: `go test ./...` (unit + integration), no frontend test framework (manual PWA/responsive testing)
**Target Platform**: Linux/Alpine container (Docker), UNRAID NAS
**Project Type**: Web service (single-page app + REST API)
**Performance Goals**: Notes list loads < 2s with 1,000 notes; search returns results < 1s
**Constraints**: Docker image < 100MB (target ~20-25MB); CGO_ENABLED=0 (pure Go build); port 8080; offline read-only via PWA service worker
**Scale/Scope**: Single-user personal app; ~1,000 notes; ~50MB file storage

**Structure Decision**: Web application layout with separate `backend/` and `frontend/` directories. The Go build embeds `frontend/` assets using `//go:embed all:../../frontend` into the binary, resulting in a single executable that serves both the API and the SPA. No separate frontend build step (no npm/webpack) вЂ” raw HTML/CSS/JS files are served directly.

## Implementation Phases

### Phase A: Foundation (P1 вЂ” Core Notes CRUD)

Delivers: create, edit, delete, view notes in Markdown. MVP.

1. Initialize Go module + chi + modernc.org/sqlite dependencies
2. Implement SQLite connection with WAL mode + FK enforcement + migration runner
3. Write `001_initial.sql` migration: `notes` table + FTS5 virtual table + triggers
4. Implement `db/notes.go`: CRUD operations + list with ordering (pinned first, then by date)
5. Implement `models/note.go` + `services/notes.go` (hashtag extraction regex)
6. Implement `handlers/notes.go`: GET /api/notes, POST, GET /:id, PUT /:id, DELETE /:id
7. Implement `frontend/index.html` + `app.css` (CSS variables for themes, responsive grid)
8. Implement `frontend/js/editor.js` with EasyMDE + 2s debounce auto-save + save button
9. Implement `frontend/js/notes.js`: list render, create, edit, delete, Marked.js preview
10. Implement `handlers/static.go` with `//go:embed` serving the SPA
11. Write `Dockerfile` (multi-stage) + verify image size < 30MB

### Phase B: Hashtags & Search (P2)

Delivers: hashtag sidebar, click-to-filter, text search.

1. Add `hashtags` + `note_hashtags` tables to migration + FTS5 sync triggers
2. Implement `db/hashtags.go` + `db/notes.go` FTS5 search query
3. Implement `handlers/hashtags.go`: GET /api/hashtags
4. Add query param filtering to GET /api/notes (`?q=` and `?hashtag=`)
5. Implement `frontend/js/hashtags.js`: sidebar with click filter, active state highlight
6. Add search input to notes list view with debounced fetch on input

### Phase C: Pin + Hashtag Management (P3 + P5)

Delivers: pin/unpin notes, rename/delete hashtags.

1. Add `pinned` field handling to GET /api/notes ordering (already in schema)
2. Implement `handlers/notes.go` PUT /:id/pin endpoint
3. Implement `handlers/hashtags.go` PUT /:name (rename) + DELETE /:name
4. Implement `services/hashtags.go`: text replacement across note content on rename/delete
5. Implement pin toggle UI (pin icon on note card) + hashtag manager modal

### Phase D: File Attachments (P4)

Delivers: upload images/docs, inline display, delete attachments.

1. Add `attachments` table to migration
2. Implement `db/attachments.go` + `services/files.go` (UUID filename, 50MB limit, MIME check)
3. Implement `handlers/attachments.go`: POST upload, DELETE, GET /files/* static serve
4. Implement `frontend/js/attachments.js`: drag-and-drop + click upload, inline image inject, PDF links
5. Handle cascade delete: when note deleted, delete physical files in `services/files.go`

### Phase E: PWA + Theme + Docker Packaging (P6)

Delivers: dark/light theme, PWA install, UNRAID-ready Docker image, GitHub Actions.

1. Implement `frontend/js/theme.js`: CSS class toggle on `<html>`, localStorage persist
2. Write `frontend/manifest.json` with app icons, `display: standalone`, `theme_color`
3. Write `frontend/sw.js`: service worker with cache-first for app shell, stale-while-revalidate for GET /api/notes, network-only for writes
4. Verify PWA installable on iOS and Android (Lighthouse audit)
5. Write `.github/workflows/docker-publish.yml`: trigger on push to `main`, multi-arch build (amd64/arm64 for UNRAID), publish to `ghcr.io/edalcin/notes`
6. Final Docker image size verification (target < 30MB)

## Key Architectural Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Backend language | Go 1.23 | Single static binary; ~8-12MB; no runtime; ideal for Alpine Docker |
| SQLite driver | modernc.org/sqlite | Pure Go (CGO_ENABLED=0); simpler Docker build; no gcc in builder |
| HTTP router | chi | Stdlib-compatible; lightweight (~1.5MB); composable middleware |
| Frontend | Vanilla JS + EasyMDE | No build step; no framework overhead; ~200KB total |
| Asset delivery | `//go:embed` | Single binary; no separate file server; simplest Docker setup |
| PWA offline | Read-only (cache-first reads, network-only writes) | Avoids conflict resolution complexity |
| Concurrent edit | Last-write-wins | Sufficient for single-user; no sync protocol needed |
| Save mechanism | Auto-save (2s debounce) + explicit save button | Belt-and-suspenders; safe on mobile |
| Docker base | alpine:3.19 | CA certs + timezone data included; easy to debug |
| Expected image size | ~20-25MB | Well within 100MB constraint |
